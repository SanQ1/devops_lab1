## 2. Документація по вебзастосунку

### Призначення застосунку
Вебсервіс призначений для обліку обладнання. Кожен об'єкт інвентарю містить такі поля: `id`, `name`, `quantity`, `created_at`. Застосунок підтримує роботу з базою даних MariaDB, автоматично виконує міграції при старті та вміє віддавати дані у форматах JSON або HTML залежно від заголовка `Accept`.

### API-ендпоінти
#### Перевірка стану (Health Checks):
* `GET /health/alive` - Завжди повертає `HTTP 200 OK`.
* `GET /health/ready` - Повертає `HTTP 200 OK`, якщо є успішне підключення до БД. Інакше — `HTTP 500`.

#### Кореневий ендпоінт:
* `GET /` - Очікує і віддає тільки `text/html`. Повертає сторінку зі списком усіх ендпоінтів бізнес-логіки.

#### Бізнес-логіка:
* `GET /items` - Виводить список усіх предметів в інвентарі (тільки `id` та `name`). 
    * Якщо `Accept: application/json` - повертає JSON-масив.
    * Якщо `Accept: text/html` - повертає просту HTML-таблицю.
* `POST /items` - Створює новий запис у системі обліку. Приймає параметри `name` та `quantity`.
* `GET /items/<id>` - Виводить детальну інформацію по конкретному запису в інвентарі (`id`, `name`, `quantity`, `created_at`) у форматі JSON або HTML-таблиці.


## 3. Документація по розгортанню інфраструктури

### Вимоги до віртуальної машини
* ОС: офіційний образ Debian 13.5.
* Ресурси: CPU: 1 vCPU, RAM: 1-2 GB, Disk: 10 GB.
* Мережа: Відкритий порт 80 (HTTP) для зовнішнього доступу.

### Архітектура розгортання
Усі компоненти автоматично розгортаються на одній ВМ:
* Nginx: Слухає на зовнішньому порту `0.0.0.0:80`, логує запити та перенаправляє трафік всередину системи на порт 8000.
* Systemd Socket Activation: Керує сокетом на `127.0.0.1:8000`. Застосунок `mywebapp` запускається через Gunicorn від імені ізольованого користувача `app` тільки тоді, коли на порт приходить запит.
* База даних: MariaDB працює локально на порту `127.0.0.1:3306`. Доступ обмежений лише зсередини ВМ.

### Користувачі в системі
Скрипт автоматично створює таких користувачів із паролем за замовчуванням `12345678`:
* `student` - Користувач для роботи, має права `sudo`.
* `teacher` - Користувач для перевірки, має права `sudo`.
* `app` - Системний користувач без оболонки для безпечного запуску сервісу.
* `operator` - Користувач для керування сервісом. Через `sudo` має право виконувати тільки команди: `start`, `stop`, `restart`, `status` для `mywebapp` та `reload` для `nginx`.

## 4. Як запустити автоматизацію розгортання

1.  Зайдіть на чисту віртуальну машину.
2.  Склонуйте цей Git-репозиторій у будь-яку папку.
3.  Перейдіть у директорію проєкту та виконайте скрипт розгортання:
    chmod +x script.sh
    sudo ./script.sh

Скрипт автоматично встановить пакети, створить базу даних, налаштує користувачів, конфігурацію Nginx, Systemd-сокети та створить файл контролю варіанта `/home/student/gradebook`.


## 5. Інструкція з тестування системи

Після виконання скрипта, працездатність системи перевіряється за допомогою таких кроків:

1.  Перевірка Health Checks:
    curl -i http://localhost/health/alive
    curl -i http://localhost/health/ready
    
    Обидва запити мають повернути статус `200 OK`.

2.  Перевірка Socket Activation:
    Зупиніть сервіс вручну: `sudo systemctl stop mywebapp.service`. Перевірте статус — сервіс має бути `inactive (dead)`. Виконайте запит `curl http://localhost/health/alive`. Запит пройде успішно, а `systemd` автоматично запустить сервіс знову.

3.  Тестування контенту (Accept header):
    * Створити предмет:
        curl -i -X POST -H "Content-Type: application/json" -d '{"name":"Switch", "quantity":5}' http://localhost/items
        
    * Перевірка HTML-таблиці:
        curl -i -H "Accept: text/html" http://localhost/items
        
    * Перевірка JSON-відповіді:
        curl -i -H "Accept: application/json" http://localhost/items
        
    * Дательна інформація про певний предмет:
        curl -i http://localhost/items/1
